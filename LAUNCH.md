# LAUNCH.md — 轻衡上架路线图

> 2026-07 与 Claude 对齐的上线/上架决策记录。目标:App Store + Play Store。
> 背景:Jiin 是马来西亚人,合规按马来西亚 PDPA,不涉及中国 ICP/App 备案。
> 原则排序:**验证 > 数据架构 > 客户端技术 > 后端选型**。别倒着做。

---

## 已定决策(含理由,别在未来会话里重新纠结)

| # | 决策 | 理由 |
|---|---|---|
| D1 | **数据本地优先、无账号体系** | 健康数据在 PDPA 下属敏感个人数据;不建服务端存储 → 合规负担趋零、隐私成卖点、商店的"账号删除"要求自动满足。多设备同步是被放弃的代价,想清楚了 |
| D2 | **不用 Firebase** | Firebase 只在"云账号"路线下才需要。选它 = 主动背上敏感数据的全部服务端义务。它是实现选项,不是战略 |
| D3 | **AI 功能走无状态代理** | 本地优先 ≠ 断网主义,承诺的是"不存储"不是"不传输"。Cloudflare Worker 十几行贴 API key 转发,不落盘不留正文日志。显式开关、默认关、开启时弹一次说明 |
| D4 | **先 Play(TWA 套壳),iOS 等信号** | TWA 对现有 PWA 几乎零改动;iOS 需 Capacitor/Flutter + 过 4.2,成本高一个量级,信号到了再花 |
| D5 | **暂不 Flutter 重写** | 重写本身不难(逻辑 2-3 会话可移植,难在 UI 手感验收轮次),但它只是 iOS 门票+长期维护性,产品验证前是提前消费。若做:第一步只移植纯函数+35 条断言,沉没成本最小 |
| D6 | **Gist 同步降级为高级选项** | "用户自备 token"对公众不成立;dsKey 直连同理,只能作隐藏高级选项 |

## 阶段 0:公网验证(本周可开始,~1 周)

**回答的问题:有没有人要用。不通过则后面全省。**

- [ ] localStorage → IndexedDB + `navigator.storage.persist()`(数据丢不起,这步不能省;迁移用现有导出/导入)
- [ ] SW 更新改为"检测到新版本 → 弹提示点击刷新"(防 v29 事故在公网复发,公网用户不会主动强刷)
- [ ] `git log -p | grep -i token` 扫提交历史,确认密钥从未入库
- [ ] 部署 Cloudflare Pages(免费、自动 HTTPS、大马可达性好);加 `_headers`:严格 CSP(`default-src 'self'`)、nosniff、Referrer-Policy
- [ ] GitHub Actions:push → `node --check` → Playwright 跑 test.html 断言 → 部署(现有人肉流程自动化)
- [ ] 极简错误上报:`onerror`/`unhandledrejection` → 十行 Worker 收日志
- [ ] 换域名后 localStorage 不跟随——迁移写进验收清单
- [ ] 发给 50 个种子用户(健身社群/Reddit/Lowyat/小红书)

**通过标准:第二周仍在用的人数 + 主动提需求的人数。数字自己定,但要事先定。**

## 阶段 1:产品化改造(验证通过后,2-4 周)

- [ ] AI 代理 Worker:无状态转发、速率限制锁死(防刷,token 成本记我账上)、免费限次(如每周 3 次周报)
- [ ] AI 开关 UI:默认关,首次开启弹说明(发送内容:匿名的餐食/训练数字;不存储;服务商有不训练承诺)
- [ ] 隐私政策页(英文必须,马来文/中文加分):本地优先架构下一页可写完,公开 URL(两商店都强制)
- [ ] Gist 同步 & dsKey 移入"高级"折叠区

## 阶段 2:Play Store(2 周,~RM300)

- [ ] 账号二选一:**SSM Enterprise + D-U-N-S + 组织账号**(豁免测试要求,推荐,D-U-N-S 有等待期尽早办)/ 个人账号 + 12 名测试者连续 opt-in 14 天(种子用户复用)
- [ ] Bubblewrap 打 TWA 包;`assetlinks.json` 域名验证
- [ ] Play App Signing 托管签名密钥,本地只留 upload key
- [ ] **Health apps declaration(强制,健身/体重/营养全命中)**:如实填,本地优先架构下多数项为"不收集";声明、app 行为、隐私政策三方必须一致,不一致会下架
- [ ] Data safety 表单、内容分级、target API level 对齐当年要求
- [ ] 商店素材:截图、图标、三语描述(EN/中/BM)
- [ ] staged rollout:5% → 20% → 100%

## 阶段 3:App Store(触发后才启动,6-8 周,$99/年)

**触发条件(写死,防止提前烧钱):Play 月活 ≥ 200,或 ≥30 人明确要 iOS。**

- 路线:Capacitor 壳 + **HealthKit 集成**(读体重/步数喂自校准 TDEE)——既是过 Guideline 4.2(Minimum Functionality,纯壳高发被拒区)的正当理由,也是真功能
- 触发 5.1.3 健康数据条款:不得用于广告/数据挖掘
- App Privacy 营养标签逐项申报;若加第三方登录必须同时提供 Sign in with Apple(无账号架构则不涉及)
- 预期被拒 1-2 次属正常,按意见改+申诉
- Phased Release 7 天爬坡;商店 app 无真回滚,只能加急发新版(expedited review)
- **此时重新评估 D5**:若决定长做 3-5 年,Flutter 重写可能优于 Capacitor,当时再判

## 长期义务(上架后按年计)

- Apple $99/年;Play target API 每年跟进;健康类政策逐年收紧(2026-01 起 Play 有医疗器械标签/Health Connect 用途说明等新要求),每年过一遍
- PDPA 2024 修正案(2025-06 生效):数据泄露 72 小时内通报 Commissioner——本地优先架构下可上报面极小,但 AI 代理若出事也算,流程要预演一次;用户量大了留意 DPO 门槛
- 商店评论要回;OS 新版适配

## 背景事实(2026-07 查证)

- Play 个人账号(2023-11 后注册)上生产前需 12 测试者×14 天连续 opt-in;组织账号豁免
- Play 健康声明表单对所有健身类 app 强制
- Apple 4.2 明确拒"与 Safari 浏览无显著差异"的壳 app;需离线可用+推送+至少一项原生能力
- 马来西亚 PDPA(Amendment)Act 2024 关键条款 2025-06 生效:72h 泄露通报、大规模处理需 DPO、数据可携权;健康数据属敏感类,处理需明确同意
